\section{Preliminary Repository Snapshot}

The repository currently contains two result sets that are useful for shaping the final analysis chapter:

\begin{itemize}
\item a repaired breadth-oriented sweep artifact, which compares several configuration families but only has one repetition per configuration; and
\item a merged decoder-depth artifact, which compares five configurations on the \texttt{config\_decoder\_test} stripped and packed variants with three repetitions each.
\end{itemize}

These artifacts are useful because they expose failure modes, refine the reporting stack, and help identify which workflow variables deserve deeper study. They should not, however, be written up as the final answer to the thesis research question without a separate held-out evaluation. In particular, the breadth sweep is descriptive rather than statistically strong because each configuration only appears once.

Its role in the thesis should be to show how the harness was used to explore the configuration space, surface failure modes, and choose promising follow-up conditions before any held-out malware-facing evaluation.

\begin{itemize}
\item This stage should be described as a broader realism check over task categories and workflow families, not as proof of general external validity.
\item It is appropriate to use this stage to motivate which interventions were carried into the focused depth study and later held-out evaluation.
\item It is not appropriate to treat this stage alone as evidence that the system generalizes to real malware families.
\item If a manually supplied malware-family pilot is run before the final locked evaluation, it should also be framed here as pre-final and methodology-shaping rather than thesis-conclusive.
\end{itemize}

\section{Focused Decoder Development Study}

The deeper \texttt{config\_decoder\_test} study should serve as a focused development case study rather than just another row in the broad sweep. Its purpose is to test whether the selected interventions appear to improve depth of understanding on a deliberately richer executable, not to replace the final held-out evaluation.

\begin{itemize}
\item Compare stripped versus packed-and-stripped variants.
\item Compare default analysis with focused configuration-recovery tasks.
\item Analyze whether the selected variable family improves depth of understanding or only broad summary quality.
\end{itemize}

Provisional observations from the current combined decoder-depth experiment:

\begin{itemize}
\item Baseline mean score is \texttt{79.232} with task-success rate \texttt{0.963}.
\item The no-role worker-prompt condition appears to be the strongest repeated-trial variant in this focused study: mean score \texttt{83.597}, score delta \texttt{+4.365}, lower mean runtime, and nearly flat cost relative to baseline. This is promising, but it remains a case-study result rather than the final thesis endpoint.
\item The single-generalist worker profile also improves mean score to \texttt{82.671}, but its task-success rate drops slightly from \texttt{0.963} to \texttt{0.926}. The current interpretation is therefore a tradeoff, not an unconditional improvement.
\item The validator-heavy pipeline appears to perform poorly in this study: mean score \texttt{3.571}, task-success rate \texttt{0.037}, validator-blocked rate \texttt{0.963}, and much larger runtime and cost than baseline. This is useful as an early warning about workflow design, not just as a low score.
\item Task-level outputs suggest that the packed UPX variant remains methodologically informative: for example, the no-role worker-prompt condition improves the packed default-analysis task by \texttt{+10.713}, while the single-generalist worker profile improves the packing triage task but regresses the parser-validation audit task.
\end{itemize}

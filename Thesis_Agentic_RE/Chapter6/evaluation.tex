\chapter{Evaluation Methodology and Experimental Design}
\label{sec:evaluation}

This chapter explains how the thesis evaluates the system. The evaluation methodology is grounded in the implemented binary-analysis harness and is designed to keep claims aligned with the available evidence. It is also structured to answer the three research questions introduced in Chapter~\ref{sec:intro}: what capability gaps remain when the system analyzes obfuscated binaries, which workflow variables materially affect analysis performance, and whether completed analyses are properly backed by artifact evidence.

\section{Evaluation Goals}

The evaluation has three goals. The first is to characterize the system's capability gaps on obfuscation-heavy binaries, including where it succeeds, where it fails to complete analysis cleanly, and where its coverage remains shallow or brittle. The second is to identify which workflow variables materially change analysis performance relative to the maintained default workflow used throughout the experiments. The third is to test whether completed analyses are grounded in proper artifact evidence rather than only plausible narrative.

\section{Benchmark Corpus and Task Model}

The binary benchmark uses an experimental corpus of authored executables and executable variants. The evaluation unit is a sample-task pair rather than only a sample. This detail is important because it lets the thesis distinguish broad understanding tasks from focused recovery tasks. That design differs from benchmark-centric evaluations such as BinMetric \cite{shang2025binmetriccomprehensivebinaryanalysis}, which are primarily organized around fixed binary-analysis question sets for model comparison. Here the target of evaluation is the configured analysis workflow acting on a sample-task unit, with intermediate artifacts and operational failures preserved alongside the final score.

Sample variants cover conditions such as stripping, packing, deceptive strings, anti-analysis behavior, and configuration decoding. Tasks include default analysis as well as focused recovery questions such as config extraction, decode-flow recovery, packing triage, and behavioral reconstruction. The corpus supports both breadth-oriented sweeps across many executables and depth-oriented studies on selected executables such as \texttt{config\_decoder\_test}. That mix is especially important for RQ1, because it stresses the system on binaries where relevant evidence is intentionally hidden, staged, or structurally awkward rather than only on cleaner structural-analysis cases.

\section{Single Runs, Sweeps, and Repetitions}

The harness supports both single-run evaluation and experiment sweeps. Sweeps are baseline-first and primarily one-variable-at-a-time, which shapes what kinds of causal claims the experiments can support.

A global baseline defines the default workflow configuration against which all variants are compared. In this thesis, that comparison point is best understood as an \emph{inferred baseline}: the maintained default workflow preserved across the repository and retained after development-stage screening, rather than a separately implemented one-agent strawman. In the maintained experiments, that baseline is a tool-complete staged static-analysis workflow with a light preparatory pass, deterministic presweeps, a balanced specialist worker mix, default prompt surfaces, and final reporting. Variant families then change one configuration surface at a time, and some families insert local family baselines where a fair comparison requires it. Repetitions support stronger claims about stability and significance. The evaluation artifacts include a repaired breadth sweep with one repetition per configuration and decoder-depth and final studies with three repetitions per configuration.

This baseline choice bounds the claims that RQ2 can support. The experiments can identify which tested workflow variables outperform or underperform the maintained default, but they do not by themselves prove that explicit orchestration is better than every simpler agent design that could have been implemented.

For reproducibility, each experiment family is defined as a self-contained launch preset. The exact preset identifiers, launch commands, and artifact locations are collected in Appendix~\ref{app:reproduction-commands} so that the body chapter can focus on methodology rather than command syntax. The harness also separates preflight readiness artifacts from executed experiment outputs to keep the experiment record reserved for completed sweeps.

\section{Broad Coverage and Final Held-Out Malware Evaluation}

The evaluation design serves three distinct roles. Development and pilot runs on the authored benchmark are used to debug workflow behavior and identify promising interventions. Broader malware-facing coverage checks provide a realism test over manually supplied families. The final held-out evaluation run carries the main empirical claims in this thesis.

The evaluation also includes a blind final-evaluation corpus. This corpus is intentionally evaluator-backed rather than a self-describing live-malware collection: the runtime sees only neutral sample aliases and task prompts, while the evaluator-facing reference set preserves family labels, source files, and scoring anchors. This separation matters both for safety and for scientific clarity. The broad-coverage pilot remains methodology-shaping, while the final held-out run carries the strongest empirical claims.

The held-out family set comprises Emotet, QakBot, TrickBot, WannaCry, and PlugX simulations, exposed to the runtime only as \texttt{sample1.exe} through \texttt{sample5.exe}. The final experiment runs one blind default-analysis task per sample across the baseline and a small set of development-informed comparison conditions. The comparison set is intentionally narrow: worker-subagent structure, worker-prompt shape, response scope, and architecture preset were varied because they proved more informative in development runs than obvious handicaps such as reduced tool access. Evaluator-side reference files anchor the configured LLM judge on sample-specific behaviors, evidence anchors, and overclaim boundaries without exposing family identity to the runtime agent.

A methodological caveat emerged only once the final held-out corpus was exercised at full scale. The final samples were materially more complex than the authored experimental benchmark, and that difference exposed a host-worker context-bloat failure mode that did not dominate the earlier development runs. In the final experiment, some Ghidra-heavy worker assignments exceeded the model input-context limit after large tool outputs and shared-state context were combined into a single worker request. This is important for interpreting the held-out results: part of the observed degradation is a capability-gap and workflow-scaling limitation, not only a reasoning-quality limitation. Chapter~\ref{sec:results} therefore reports final performance alongside per-executable failure distributions rather than only aggregate score summaries.

\section{Baselines and Experimental Variables}

The harness can express several variable families, but the thesis experiments focus on those that survived development-stage screening: response scope, worker subagent profile, worker prompt shape, worker role-prompt mode, architecture preset, pipeline preset, and selected validator-backed pipeline follow-ups in the decoder-depth study. Appendix~\ref{app:sample-variant-traceability} gives the complete mapping from variant labels to baseline-relative changes.

Prompt-shape and role-prompt changes are treated here as empirical variables rather than assumed improvements. That choice follows from both the archive-stage screening results and from external evidence that persona-style prompting does not reliably improve factual accuracy even when it changes response style \cite{basil2025promptingscience4}.

\section{Judging and Scoring}

Run outputs are judged by a configured LLM-based evaluator under a structured binary-analysis rubric and then converted into continuous scores. In the maintained thesis runs, this is an LLM-as-a-judge procedure with an explicit judge model and prompt, not expert human annotation or objective ground truth. The judging methodology matters because it defines how the experiments distinguish weak answers from unsupported or infrastructure-dominated failures.

Per-task LLM-judge outputs preserve structured feedback and an overall score. Legacy pass-or-fail thresholds still exist, but sweep analysis focuses on score deltas, success rates, error rates, timing summaries, and tool-use summaries rather than binary outcomes alone. The harness preserves analysis outcomes such as completed, analysis error, no result, judge error, and validator blocked. The final held-out evaluation scaffold also includes evaluator-side reference templates checked into the repository. These references anchor the LLM judge's evidence expectations, supported-behavior boundaries, and false-positive controls rather than merely family attribution. This matters directly for RQ3: the judge does not only score task completion, but also dimensions such as \texttt{evidence\_grounding}, \texttt{specificity}, and \texttt{false\_claim\_control}, which let the thesis ask whether agents actually support claims with visible artifacts.

\section{Statistical Treatment and Comparison Strategy}

The analysis in this thesis does not rely only on descriptive movement. The harness supports paired comparisons and limited significance analysis when repetitions or enough matched task pairs exist.

Where repeated runs exist, the harness supports replicate-level comparison. For matched sample-task pairs, it supports paired-task comparison. It reports score deltas, task success deltas, and significance outputs where the data support them. Throughout, the analysis maintains an explicit distinction between descriptive one-off sweep results and repeated experiments that can support stronger stability claims.

\section{Mapping Experimental Evidence to the Research Questions}

The main evidence types in the thesis map directly to the three research questions. RQ1 relies most heavily on the held-out sample descriptions, task-success rates, failure accounting, unrecovered-versus-repaired comparisons, timing summaries, and repeated-tool-use summaries, because those artifacts show where the system remains brittle or shallow on obfuscation-heavy binaries. RQ2 relies on baseline-first variant comparisons, overall score deltas, per-executable heatmaps, and significance summaries, because that question is about how changing the workflow alters performance. RQ3 relies on the LLM-judge rubric dimensions, preserved LLM-judge narratives, and selected run artifacts, because that question asks whether the final claims remain visibly tied to recovered evidence.

This mapping also bounds the kinds of claims each chapter can make. Aggregate ranking figures are most useful for RQ2, but they are not by themselves sufficient for RQ1 or RQ3. Likewise, LLM-judge evidence-grounding scores matter strongly for RQ3, but they do not by themselves identify which workflow variable should be preferred under RQ2. Chapter~\ref{sec:results} therefore keeps the subsection order aligned with the research questions so that each evidentiary claim is made against the artifacts most relevant to it.
